Sudoku-Erlang

Sudoku Interpreter w/ Erlang

Sudoku Interpreter implemented by Erlang.

Project purpose

  • Build tools solving multiple problems about sudoku
  • Learn basic Erlang syntax & how to build Erlang program

Modules

  • sudoku: reading board + inspect sudoku’s properties
  • knuth: to solve “exact cover” problems, including sudoku
  • testcase: “exact cover” problem examples(exact cover, 2x2 sudoku, 9x9 sudoku)
  • solver: sudoku solver using module “knuth.erl”

How I used it

  1. Add sudoku problem into “./data” folder with format .csv
  2. Call “solver:solve/1” or “solver:solve/2” with path of sudoku problem

How can I test it?

Use “solve_sudoku.escript” to test sudoku solver.

프로그램 설명

한국IT직업전문학교에서의 2018 G-STAR 출품 게임 “Kingsnake”의 서비스를 위해 개발 된 프로그램입니다.

  • 프로젝트 기간: 약 7개월(2018/05 ~ 2018/11)
    • 구조 설정: 1month
    • 게임 서버: 3month
    • DB 서버: 2month
    • 버그 수정: 1month
  • 사용된 기술
    • C++(MSVC)
    • STL(C++11, Container 위주)
    • 멀티스레드
    • 네트워크(TCP/IP + IOCP)
    • MySQL

기여한 파트

  • 통신 구조 설정
    • 클라이언트-게임서버-DB서버
    • 통신 모듈(SOCKET&buffer + SERVER)
      • IOCPClient h / cpp
      • IOCPServer h / cpp
    • 통신 방법(프로토콜) 및 데이터 직렬화/마샬링
      • IOCPClient h / cpp
      • protocol
  • 유저 정보 저장 및 전달
    • MySQL로 저장 및 불러오기
      • MYSQLManager h / cpp
    • 회원가입, 가입탈퇴
    • 유저 정보 변경
      • 캐릭터, 아이템 세팅
      • 아이템 추가
    • DatabaseManager h / cpp
  • 게임 진행에 필요한 정적 데이터 로딩
    • CSV 형식으로 저장, 기획자로부터 제공받은 후 CSVReader로 읽어옵니다.
    • CSVReader h / cpp

(미리 알았으면 좋았을)보완점들

  • C++/IOCP로 작성 됨
    • 너무 과한 기술이 아니였나 하는 후회..
    • Python/nodeJS + REST API를 미리 알았더라면
      • 빠른 코딩 속도
      • 많은 레퍼런스
    • SELECT 모델을 사용 했더라면
  • code에 작성된 SQL Query
    • Query 변경 시 빌드를 다시 해야 함
    • .sql 파일로 추출 후 불러오기 하는 방안
  • DatabaseManager에서 모든 기능을 담당
    • 유저정보, 아이템 정보, 정적데이터
    • 각 분야를 담당하는 별개의 객체들을 구성(RegisterData, SettingData, StaticData 등..)
  • 직렬화 방법의 스크립트 화
    • 데이터 직렬화 방법은 수시로 바뀔 수 있음 => 바뀔 때마다 새로 빌드
    • CSV, JSON 등으로 직렬화 방법을 정의 => 방법대로 직렬화 후 GameServer에게 전송
      • 나아가, GameServer와 DBServer의 직렬화 방법을 검증한다면?
  • 정적 데이터의 저장 공간 변경
    • 정적 데이터(맵 크기, 캐릭터 공격력 등 개발자가 정하는 데이터)는 자주 요구된다
    • 메모리에 저장 할 수 있었더라면(StaticDataManager로 전부 저장)
    • NoSQL의 Key-Value DB(Redis 등)를 사용 했더라면
      • RDBMS(MSSQL, MySQL) 사용 경험밖에 없어서, 해봤다면 큰 경험이 되었을 것

    • 구조 설정: 1month
    • 게임 서버: 3month
    • DB 서버: 2month

    • C++(MSVC)
    • STL(C++11, Container 위주)
    • 멀티스레드
    • 네트워크(TCP/IP + IOCP)

기여한 파트

  • 통신 구조 설정
    • 클라이언트-게임서버-DB서버
    • 통신 모듈(SOCKET&buffer + SERVER)
      • IOCPClient h / cpp
      • IOCPServer h / cpp

    • 데이터 검증(암호화/복호화)
      • EncryptManager h / cpp

    • Log-in 정보
    • Lobby 화면 정보(닉네임, 레벨, 경험치, 재화)

      • GameObject h / cpp

(미리 알았으면 좋았을)보완점들

  • 약간의 변경에도 빌드 후 테스트 가능
    • 변경이 자주 일어나는 부분의 Script화(예를들어 Lua라던가…)
    • 정적 데이터(맵 크기, 공격력, 게임 진행 시간 등)는 파일/DB 사용(일부분은 적용 됨)
  • STL의 사용 범위
    • 너무 map, list 등의 Container에만 치중 됨
    • 스마트 포인터, std::thread, type_traits 등 적용해볼 사안이 많아보임
  • 디자인 패턴의 부재
    • Observer, State 등 알아두면 편했을 패턴이 많다.
  • 통신 모듈의 라이브러리화
    • 네트워크 통신 부분만 떼어내서 다른 프로젝트에 활용 할 수 있지 않을까
    • 버퍼 크기, 통신 프로토콜(TCP, UDP), 네이글 알고리즘 적용 여부 등을 정의하는 config 파일
    • 나아가서 서버 엔진(Gamesparks, Photon)을 만들어 보는건…?
    • 이와는 별개로, 클래스들의 사용법을 적어두는 설명서(wiki?)도 만들었다면…
